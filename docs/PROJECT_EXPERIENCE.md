# AI Agent 项目经验与能力总结文档

> **文档定位：** 从全栈项目交付实践中提炼的通用方法论。面向 AI Agent，提供从项目分析、技能调用、功能开发到问题排查的全链路指导，适用于任何技术栈的全栈项目。

---

## 一、项目分析与理解方法论

### 1.1 系统化需求分析框架

遵循 **"文档驱动 → 代码验证 → 差距清单"** 三阶段框架：

**阶段一：文档驱动** — 按优先级阅读：需求规格 → 数据库设计 → 技术架构 → API 文档 → 部署文档

关键要点：
- 文档间矛盾以最新版本为准
- 文档不完整时通过代码逆向补充
- 关注"TODO"、"暂不实现"等标记

**阶段二：代码验证** — 对比文档与代码实际状态：

| 检查维度 | 检查方法 | 典型发现 |
|---------|---------|---------|
| 功能完整性 | 按功能清单逐项测试 | 部分功能缺失或半成品 |
| 数据库一致性 | 对比建表脚本与文档 | 表结构/字段缺失或变更 |
| API 对齐 | DevTools 查看实际请求 | 前端调用的接口后端不存在 |
| 代码质量 | 阅读核心模块源码 | 明显 bug、硬编码、安全漏洞 |
| 配置正确性 | 检查配置文件 | 端口冲突、路径遗漏 |

**阶段三：差距清单** — 按优先级分类：P0 阻塞性 → P1 核心缺失 → P2 体验问题 → P3 增强功能

### 1.2 项目背景调研与目标拆解

**背景调研四问法：**

| 问题 | 为什么重要 |
|------|-----------|
| 项目性质是什么？ | 决定技术选型的保守/激进程度 |
| 目标用户是谁？ | 决定功能优先级和 UI 复杂度 |
| 运行环境是什么？ | 决定部署方案和性能要求 |
| 验收标准是什么？ | 决定质量底线和交付范围 |

**范围界定：** IN-Scope（文档明确要求）、OUT-of-Scope（文档标注暂不实现）、Gray Zone（未明确需确认）

### 1.3 复杂度评估

对每个功能模块评估：数据库变更量、后端工作量、前端工作量、集成风险。决策规则：低复杂度可并行，高复杂度串行验证，高集成风险先写测试。

### 1.4 关键干系人期望管理

- 用户提出模糊需求 → 主动提供 2-3 个具体选项
- 发现文档矛盾 → 及时确认而非自行假设
- 大范围修改前 → 先说明方案获得认可

### 1.5 技术栈快速理解框架

面对任何不熟悉的技术栈，按六维建立理解模型：

| 维度 | 关键问题 | 通用答案模式 |
|------|---------|------------|
| 版本兼容 | 框架大版本有何破坏性变更？ | 查 Release Notes / Migration Guide |
| 配置方式 | 框架如何读取配置？ | 环境变量 / 配置文件 / 代码配置 |
| 数据流 | 请求从入口到响应经过哪些层？ | 路由 → 中间件 → 控制器 → 服务 → 数据层 |
| 认证机制 | 如何识别和授权用户？ | Session / JWT / OAuth / API Key |
| 错误处理 | 异常如何传播和展示？ | 全局异常处理器 / 中间件捕获 / 统一响应格式 |
| 前后端契约 | API 如何约定？ | RESTful / GraphQL / RPC + 统一响应封装 |

### 1.6 数据库逆向分析法

当文档不完整时，通过数据库表结构反推业务逻辑：

| 数据库特征 | 反推的业务逻辑 |
|-----------|--------------|
| `status` 整型字段 | 状态机（需确认每个值的含义和流转规则） |
| `UNIQUE INDEX (a, b)` | 去重需求（这两个字段的组合应唯一） |
| `JSON` 类型字段 | 灵活数据结构，无需关联表 |
| `is_default` 布尔字段 | 唯一默认项（同用户只能有一个默认） |
| `ON DUPLICATE KEY UPDATE` | 幂等写入（重复访问更新而非新增） |
| `DECIMAL` 精确数值 | 精确金额，不可用浮点数 |

## 二、关键技能(Skills)清单与调用指南

### 2.1 Agent Skills 调用清单

以下 Skills 是 Agent 在项目各阶段可直接通过 Skill 工具调用的能力模块。调用方式：`Skill(name="技能名")`，调用后技能会展开为详细指令指导 Agent 执行。

| Skill 名称 | 来源 | 调用时机 | 项目阶段 | 说明 |
|------------|------|---------|---------|------|
| `web-dev` | Trae 内置 | 从零创建前端页面/组件 | 阶段 0-4 | 生成高质量前端界面，包含设计规范和代码模板 |
| `frontend-design` | vercel-labs/agent-skills | 优化现有 UI 设计质量 | 阶段 4 | 审查并提升界面设计，避免通用 AI 美学 |
| `webapp-testing` | vercel-labs/agent-skills | 验证前端功能、调试 UI | 阶段 5 | Playwright 交互测试、截图、浏览器日志 |
| `TRAE-code-review` | Trae 内置 | 审查代码质量、合并请求 | 阶段 3-5 | 结构化代码审查，输出质量/正确性反馈 |
| `TRAE-security-review` | Trae 内置 | 扫描安全漏洞 | 阶段 5 | 审查代码安全性，输出风险与最佳实践 |
| `test-driven-development` | obra/superpowers | 实现功能前先写测试 | 阶段 2-3 | TDD 流程：先测试 → 再实现 |
| `web-design-guidelines` | vercel-labs/agent-skills | 审查 UI 是否符合规范 | 阶段 4 | 检查可访问性、设计最佳实践 |

**调用规则：**
1. 用户意图匹配 Skill 描述时，**立即调用**，不要先输出文字说明
2. Skill 展开后按其指令执行，不要跳过步骤
3. 同一时刻不要重复调用已运行的 Skill
4. 内置 CLI 命令（/help、/clear 等）不通过 Skill 调用

**项目流程中的 Skill 调用顺序：**

```
阶段 0-2: 基础搭建
  └─ web-dev          → 创建前端页面骨架和核心组件

阶段 3: 增强功能
  └─ test-driven-development → 先写测试再实现新功能

阶段 4: 体验优化
  ├─ frontend-design  → 优化 UI 设计质量
  └─ web-design-guidelines → 审查 UI 规范合规性

阶段 5: 质量保障
  ├─ webapp-testing   → Playwright E2E 测试验证
  ├─ TRAE-code-review → 代码质量审查
  └─ TRAE-security-review → 安全漏洞扫描
```

### 2.2 Skills 安装配置

#### 内置 Skills（无需安装）

Trae IDE 内置以下 Skills，开箱即用：`web-dev`、`skill-creator`、`TRAE-code-review`、`TRAE-security-review`、`TRAE-generate-mini-app`

#### 外部 Skills 安装

使用 `npx skills` CLI 安装外部 Skills。**前置条件：** Node.js 18+（含 npx）。

**安装命令格式：**
```bash
# 通用格式
npx skills add <owner/repo> --skill <skill-name> -a <agent> -g -y

# 参数说明：
#   <owner/repo>    GitHub 仓库（如 vercel-labs/agent-skills）
#   --skill         指定安装的技能名（省略则安装仓库全部技能）
#   -a <agent>      目标 Agent：Trae CN 用 trae-cn，国际版用 trae
#   -g              全局安装（所有项目可用）
#   -y              跳过确认提示
```

**按来源安装：**

```bash
# 1. Vercel 官方技能包（frontend-design, web-design-guidelines 等）
npx skills add vercel-labs/agent-skills -a trae-cn -g -y

# 仅安装特定技能
npx skills add vercel-labs/agent-skills --skill frontend-design --skill web-design-guidelines -a trae-cn -g -y

# 2. Superpowers 工作流技能（test-driven-development 等）
npx skills add obra/superpowers -a trae-cn -g -y

# 仅安装 TDD 技能
npx skills add obra/superpowers --skill test-driven-development -a trae-cn -g -y
```

**验证安装：**
```bash
# 列出已安装的技能
npx skills list -g

# 搜索可用技能
npx skills find <keyword>
```

**其他常用命令：**
```bash
npx skills update              # 更新所有已安装技能
npx skills remove <skill-name> # 移除技能
npx skills init <name>         # 创建新技能模板
```

**手动安装（备选方案）：** 如果 `npx skills` 不可用，可手动将 SKILL.md 文件复制到项目目录：
```
# Trae CN 版本
.trae-cn/skills/<skill-name>/SKILL.md

# Trae 国际版
.trae/skills/<skill-name>/SKILL.md
```

### 2.3 技术栈环境配置通用方法

面对新项目时，按以下步骤快速搭建环境：

**Step 1: 识别技术栈** — 读取项目配置文件确定技术栈：
- 后端：`pom.xml`(Maven/Java) / `build.gradle`(Gradle) / `package.json`(Node) / `requirements.txt`(Python) / `go.mod`(Go)
- 前端：`package.json` 中的 dependencies / `vite.config.*` / `webpack.config.*`
- 数据库：`docker-compose.yml` / `init.sql` / `prisma/schema.prisma`

**Step 2: 安装运行时** — 根据识别结果安装：
```bash
# Java 项目
java -version && mvn -version

# Node 项目
node -version && npm -version

# Python 项目
python --version && pip --version

# Docker 环境
docker-compose ps
```

**Step 3: 验证环境完整性** — 通用检查流程：
```bash
# 1. 数据库可连接
# 2. 后端编译通过
# 3. 后端 API 可达（curl 测试）
# 4. 前端构建通过
# 5. 前端可访问后端（代理/直连）
```

### 2.4 后端核心模式（技术栈无关）

无论使用何种后端框架，以下模式必须建立：

**统一响应封装：** 所有 API 返回统一格式 `{ code, message, data }`，前端拦截器统一处理

**认证授权体系：**
- 前后端分离 → 无状态认证（JWT / Token）
- 传统服务端渲染 → Session + Cookie
- 新增 API 后必须同步更新授权配置（公开/需认证/需角色）

**全局异常处理：** 捕获所有未处理异常，返回统一错误格式，不暴露堆栈信息

**ORM 字段映射：** 确认数据库命名风格（snake_case）与代码命名风格（camelCase）的映射规则

**JSON 字段处理：** 数据库中的 JSON 字段，ORM 层通常返回字符串，Service 层需序列化/反序列化

**事务管理：** 涉及多步写操作（如扣款+更新状态）必须使用事务

### 2.5 前端核心模式（技术栈无关）

**HTTP 请求封装：**
- 请求拦截器：自动注入认证 Token
- 响应拦截器：解包统一响应格式 + 401 自动跳转登录
- 核心：明确拦截器解包了几层，API 层实际拿到什么类型

**前后端数据流（最易出错）：**
```
后端返回: { code: 200, data: { items: "[{...}]" } }
    ↓
响应拦截器解包 → 拿到业务数据
    ↓
JSON 字段可能是字符串而非对象 → 必须解析
    ↓
类型系统可能与实际数据不匹配 → 需要类型断言/绕过
```

**路由配置必做项：**
- 页面跳转后滚动到顶部（`scrollBehavior`）
- 需认证页面的路由守卫
- 路由元信息标记（`meta.requiresAuth`）

**状态管理要点：**
- 修改后端数据后重新拉取保证一致性
- JSON 字段统一做 `typeof === 'string' ? JSON.parse : 直接使用` 处理

### 2.6 数据库核心模式

**增量更新策略：** 维护增量变更脚本，使用 `IF NOT EXISTS` 保证幂等执行

**去重模式：** 需要去重的表添加唯一索引，INSERT 改为幂等插入（如 `ON DUPLICATE KEY UPDATE`）

**添加唯一索引前：** 先清理已有重复数据，否则 ALTER TABLE 会失败

## 三、完整操作指南与流程规范

### 3.1 项目实施阶段

```
阶段 0: 环境搭建 → 可启动的空项目
阶段 1: 基础设施（统一响应 + 认证 + 异常处理）→ 登录/注册可用
阶段 2: 核心业务 → 核心流程可走通
阶段 3: 增强功能 → 功能完整
阶段 4: 体验优化（UI + 交互）→ 可演示
阶段 5: 质量保障（测试 + 修复 + 清理）→ 可交付
```

### 3.2 全栈功能开发标准流程

```
1. [DB] 设计表结构 → 写增量变更脚本 → 执行
2. [后端] 数据模型 → 数据访问层 → 业务逻辑层 → 控制器层 → 授权配置
3. [前端] API 封装 → 状态管理 → 页面组件 → 路由配置
4. [验证] 后端编译 → 前端构建 → 功能测试 → 回归测试
```

每步质量检查点：DB（表创建成功、索引存在）、后端（编译通过、API 可达）、前端（构建通过、手动测试）

### 3.3 Bug 修复五步法

1. **复现** — 确认现象、触发条件、影响范围
2. **定位** — 分层排查：网络/连接 → 认证/授权 → 业务逻辑 → 数据持久
3. **修复** — 最小化改动，不引入新功能，不重构无关代码
4. **验证** — 编译通过 + 复现步骤不再触发 + 相关功能仍正常
5. **回归** — 运行完整测试套件，确认无副作用

### 3.4 关键任务执行标准

| 任务 | 执行标准 | 验证方法 |
|------|---------|---------|
| 新增功能 | 全栈贯通（DB→后端→前端） | 端到端流程测试 |
| Bug 修复 | 根因明确、最小改动 | 复现步骤不再触发 |
| 数据库变更 | 幂等可重复执行 | 清库后重新执行不报错 |
| UI 优化 | 无溢出/重叠/截断 | 不同内容长度下验证 |
| 项目清理 | 无临时文件/死代码 | 编译 + 构建通过 |

### 3.5 Agent 协作模式

- **信息同步：** 修改后告知改了什么、发现矛盾主动提出、大改前先说明方案
- **并行策略：** 无依赖任务可并行，有依赖必须串行
- **变更追踪：** TodoWrite 记录进度，完成标记 summary，修改后列变更清单

## 四、问题解决策略与最佳实践

### 4.1 分层排查法

面对问题时，按层逐级排查，避免盲目猜测：

```
Layer 1: 网络/连接 — 服务启动？端口正确？代理生效？容器运行？
Layer 2: 认证/授权 — Token 有效？已发送？路径放行？角色匹配？
Layer 3: 业务逻辑 — 参数正确？响应格式？数据类型？业务规则？
Layer 4: 数据持久 — SQL 正确？字段映射？事务生效？数据正确？
```

**快速定位：**
- 页面空白/无数据 → Network 请求状态码决定层级
- 操作无效果 → 请求是否发出 → 响应是否成功 → 状态是否刷新
- UI 异常 → CSS flex/overflow/position
- 数据不一致 → 响应拦截器/JSON 解析/SQL 逻辑

### 4.2 通用难题与解决方案

**难题一：前后端数据格式不对齐**

症状：前端显示 `[object Object]`、无法遍历
根因：后端 JSON 字段返回字符串而非对象
解决：前端统一做类型检查 + JSON.parse
预防：每写新接口立即在 DevTools 验证返回格式

**难题二：授权配置路径遗漏**

症状：新 API 返回 403
根因：路径未配置，被兜底规则拦截
解决：新增 API 后同步更新授权配置
预防：建立新增 API 的授权检查清单

**难题三：CSS Flex 布局溢出**

症状：元素重叠、被挤掉
解决：自适应侧 `flex: 1; min-width: 0; overflow: hidden`，固定侧 `flex-shrink: 0`
预防：卡片布局必须测试不同内容长度

**难题四：唯一索引添加失败**

症状：ALTER TABLE 报 Duplicate entry
解决：先清理重复数据，再添加索引

**难题五：前端过滤 vs 后端过滤**

决策规则：数据量小且不分页 → 前端可接受；数据量大或分页 → 必须后端；搜索/排序/筛选 → 必须后端

### 4.3 各阶段避坑指南

| 阶段 | 最佳实践 | 避坑 |
|------|---------|------|
| 基础设施 | 统一响应/认证/异常处理在业务代码之前完成 | 不要跳过认证后期补 |
| 核心业务 | 先跑通核心流程再添加功能 | 多步写操作必须用事务；JSON 字段前端必须解析 |
| 体验优化 | UI 修改后截图验证；外部资源用可控来源 | 不要忽略路由滚动重置；不要用固定宽度 |

### 4.4 问题知识库

| 问题模式 | 根因 | 解决方案 | 预防措施 |
|---------|------|---------|---------|
| 数据重复不合并 | 写入无去重逻辑 | 唯一索引 + 幂等插入 | 需去重的表加唯一索引 |
| 搜索筛选无效 | 前端过滤无法处理分页 | 改后端查询过滤 | 过滤/排序一律后端处理 |
| 外部资源加载失败 | 第三方 URL 失效 | 替换为可控来源 | 不依赖不可控外部服务 |
| 导航状态不正确 | CSS 选择器与框架类名不匹配 | 检查框架实际添加的 CSS 类 | 验证框架的类名机制 |
| 状态机行为异常 | 状态值与文档不一致 | 确认每个状态值的含义 | 状态值与文档严格对齐 |
| 页面跳转位置异常 | 路由未配置滚动行为 | 配置 scrollBehavior | 新项目必须配置 |
| 布局元素重叠 | flex 布局溢出 | `flex:1; min-width:0` | 测试不同内容长度 |
| 新 API 返回 403 | 授权配置遗漏 | 同步更新路径规则 | 新增 API 检查授权配置 |

## 五、项目实施心得与核心心法

### 5.1 第一性原理

剥离非必要复杂度，只做必须做的事。技术选型三问：
1. 不用这个技术，项目能跑吗？→ 能跑就不加
2. 加了解决了什么问题？→ 问题必须真实存在
3. 引入多少复杂度？→ 复杂度收益比必须 > 3:1

### 5.2 增量演进

每个阶段交付可运行的产品。核心流程优先，增强功能其次。数据库变更用增量脚本，不修改初始建表脚本。

### 5.3 前后端对齐：最大风险源

三大对齐风险：
1. **数据格式** — 后端 JSON 字段可能返回字符串，前端期望对象 → 必须类型检查+解析
2. **响应层级** — HTTP 拦截器可能解包了响应，API 层拿到的层级与预期不同 → 明确每层返回什么
3. **字段命名** — 数据库命名风格 vs 代码命名风格 → 确认 ORM 映射规则

核心心法：**每写新接口，立即验证实际返回格式，不要假设。**

### 5.4 配置文件：隐式依赖

配置文件不在业务代码中，但影响所有功能。改完业务代码后，必须检查相关配置是否需要同步更新：

| 配置类型 | 常见遗漏 |
|---------|---------|
| 授权/安全配置 | 新增 API 未添加路径规则 |
| 路由配置 | 新增页面未添加路由 |
| 代理配置 | 后端端口/地址变更未同步 |
| 容器编排 | 服务版本/密码不匹配 |

### 5.5 测试驱动修复

代码修改必须经过验证：编译检查 → 功能验证 → 回归测试。不同修改的验证级别不同（CSS 浏览器查看即可，业务逻辑需功能测试+回归，安全配置需逐端点测试）。

### 5.6 应对变更与风险

- 增量脚本保证数据库变更可追溯
- 后端分层保证业务变更只影响局部
- 外部依赖失效 → 准备替代方案
- 版本兼容 → 锁定依赖版本
- 性能 → 先保证功能正确再优化

## 六、可迁移到其他项目的经验与教训

### 6.1 通用全栈项目架构

```
后端分层: 路由/控制器 → 业务逻辑 → 数据访问 → 数据模型
前端分层: 页面组件 → 状态管理 → API 封装 → 工具函数
基础设施: 统一响应 + 全局异常 + 认证授权 + CORS + 路由守卫
```

无论技术栈如何变化（Java/Python/Go/Node + Vue/React/Angular），这三层架构不变，只需替换具体实现。

### 6.2 项目启动必做清单

```
□ 环境: 数据库环境 + 后端骨架 + 前端骨架 + .gitignore
□ 基础: 统一响应 + 全局异常 + 认证授权 + CORS + HTTP 拦截器 + 路由守卫
□ 数据库: 建表脚本 + 测试数据 + 增量变更脚本 + 密码加密
□ 规范: API 契约确认 + JSON 字段处理策略 + 测试策略
```

### 6.3 常见陷阱速查表

| 陷阱 | 表现 | 解决方案 |
|------|------|---------|
| JSON 字符串未解析 | 显示 [object Object] | 类型检查 + JSON.parse |
| 授权配置路径遗漏 | 新 API 返回 403 | 同步更新授权配置 |
| SQL 关键字冲突 | 查询报语法错误 | 引号包裹保留字 |
| 前端过滤不一致 | 搜索结果不完整 | 改后端查询过滤 |
| 唯一索引前有重复 | 建索引报错 | 先清理重复再添加 |
| 路由滚动不重置 | 页面跳转停底部 | 配置 scrollBehavior |
| CSS flex 溢出 | 元素重叠 | `flex:1; min-width:0` |
| 明文密码存储 | 登录永远失败 | 测试数据也必须加密 |
| 字段映射错误 | 查询结果为 null | 确认 ORM 映射配置 |
| 状态机值错误 | 操作不符合预期 | 状态值与文档严格对齐 |
| Token 过期未处理 | 操作突然失败 | 401 自动跳转登录 |

### 6.4 可复用流程模板

**新增功能：** DB 建表 → 数据模型 → 数据访问 → 业务逻辑 → 控制器 → 授权配置 → API 封装 → 状态管理 → 页面组件 → 路由 → 编译验证

**Bug 修复：** 复现 → 分层定位 → 最小修复 → 编译验证 → 回归测试

**项目收尾：** 删临时文件 → 删测试代码 → 更新文档 → 验证从零启动 → 确认 .gitignore

### 6.5 典型失败案例警示

**前端过滤：** 分页场景下前端只拿到第一页数据，过滤结果不完整。警示：筛选/排序一律后端处理。

**数据重复：** 每次直接 INSERT 导致重复记录。警示：需要去重的场景优先用数据库唯一约束。

**布局溢出：** 所有元素水平排列，条件内容出现时空间不够。警示：卡片布局必须测试不同内容长度。

### 6.6 性能优化

| 方向 | 措施 | 场景 |
|------|------|------|
| 前端构建 | UI 库按需引入 | bundle 过大 |
| 数据库 | 高频查询字段建索引 | 慢查询 |
| 查询 | JOIN/批量查询替代循环 | N+1 问题 |
| 前端渲染 | 骨架屏替代 spinner | 首屏体验 |

### 6.7 技术决策原则

面对技术选型时，遵循以下原则而非绑定具体技术：

| 决策维度 | 选择原则 | 反模式 |
|---------|---------|--------|
| ORM | SQL 可控、学习成本低 | 过度抽象导致难以调试 |
| 认证 | 适合项目架构（分离→Token，SSR→Session） | 盲目追新忽略场景 |
| 缓存 | 按需引入，先验证瓶颈 | 过早优化 |
| 去重 | 数据库约束优先于应用层判断 | 应用层判断有并发风险 |
| 过滤 | 后端处理保证一致性 | 前端过滤分页场景不完整 |
